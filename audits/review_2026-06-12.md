# Project Review — 2026-06-12

A skeptical whole-project audit of the cross-asset EMS at plan completion (195/195 tasks).
Three lenses: **test coverage**, **feature reality vs. claims**, and **usability/operability**.
Findings are concrete (file paths) and prioritized. The summary first, because it changes how
to read everything below.

---

## Executive summary

The plan is "complete," but completion was measured per-task, and most post-Phase-18 tasks were
delivered as **well-tested libraries that no runtime constructs**. The single composition root —
`TraderDesktopEdgeMain` — wires roughly half the system. The other half (compliance gate with the
fat-finger check, SOR + algo wheel, TCA/best-ex, fees, surveillance, drop-copy, CAT lifecycle,
IOI, jurisdiction routing, quote multicast, venue dialects) exists, passes its unit tests, and is
reachable by nothing. **The biggest gap in this project is not missing code — it is missing
integration.**

Priority order for the next phase of work:

1. **P0 — Wire the unwired** (§2.1). A buyer demo that claims fat-finger protection while the
   demo edge performs no compliance check is a credibility risk, not a style issue.
2. **P0 — Trader safety UX** (§3.1): no confirmation on cancel, stale amend snapshots,
   native `prompt()` on the kill switch.
3. **P1 — Integration tests** (§1.3): zero `*IntegrationTest` files; the composition root has no
   test at all.
4. **P1 — Config out of code** (§3.4): venues, accounts, thresholds, TTLs are compile-time
   constants; a config service exists and is unused for them.
5. **P2 — the rest**: thin-coverage classes, formatting/i18n, docs drift.

## 1. Test coverage gaps

### 1.1 Untested classes (no test file at all)

The systematic sweep found **50+ main-source classes with no corresponding test**, concentrated
in the oldest modules and in the newest adapters:

| Area | Examples | Why it matters |
|---|---|---|
| Reg-submission adapters | `MsrbRtrsAdapter`, `CftcSdrAdapter`, `Rts22Adapter` have only the combined `RegSubmissionAdaptersTest` (payload pins) — no retry/nack-path tests through `InMemoryRegulatoryReportingService` for MSRB/SDR/RTS22 specifically | a nacked RTS 22 report that never retries is a reg breach |
| `ems-aaa` internals | session bookkeeping/log classes | identity is the root of every permission decision |
| `ems-fix-bridge` REST internals | `RestHttpServer` (only exercised indirectly), `WsEventStreamServer` partial | the production edge surface |
| Venue plumbing | `VenueAdapterRegistry`, `InMemoryVenueAdapterRegistry`, `VenueSelection`, `AbstractVenueAdapter` state transitions | the router consults these before every route |
| `BloombergFeedMain`, `FixSimulatorMain`, `TraderDesktopEdgeMain` | all `main()` entrypoints | see §1.3 — the wiring itself is untested |

### 1.2 Thin tests (happy-path only)

- `CatReporter` — batch/repair pinned, but no out-of-order capture, no batch-size limits, no
  multi-day boundary behavior.
- `FixVenueGateway` — session tests exist; the new dialect hook has one integration test
  (BrokerTec) but no per-dialect gateway round-trip vs `FixVenueSimulator`.
- Concurrency-sensitive classes (`MachineGunCheck`, `BlotterPublisher` accumulators,
  `DropCopyService`, `EspClickService`, OMS managers using `ConcurrentHashMap`/`AtomicLong`)
  have **zero concurrent/stress tests** — every test is single-threaded. The blotter WAP
  accumulator's record-before-delegate dance is precisely the kind of code that breaks only
  under interleaving.

### 1.3 The integration hole (highest-value gap)

- **No `*IntegrationTest` exists anywhere.** Components are tested in isolation; the seams
  between them are tested nowhere except implicitly by the Playwright E2E suite (which only
  covers what the demo edge wires — see §2.1).
- **`TraderDesktopEdgeMain` has no test.** It is ~700 lines of hand-wiring (the only composition
  root in the repo) and every regression in it is discovered by booting the demo.
- The E2E suite (12 specs, 425 lines) is good for what it covers but maps 1:1 to demo-edge
  features. Anything not wired into the edge has no end-to-end proof.

### 1.4 UI module coverage

| Module | Coverage | Gap |
|---|---|---|
| `rfq.ts`, `ticket.ts`, `main.ts` flows | E2E specs | covered |
| `recovery.ts` | **none** | the backend-restart watchdog was probe-verified once, manually; no repeatable test |
| `docking.ts`, `layout.ts` | **none** | layout persistence/reset regressions would ship silently |
| `aggregates.ts` (MULTI badge) | **none** in CI | probe-verified manually only |
| `grid-actions.ts` keyboard selection | partial (context-menu specs) | shift+arrows/page-keys untested in CI |
| All TS modules | **zero unit tests** | the UI has only E2E; no test runner (vitest) is configured |

## 2. Feature gaps — reality vs. the 195/195 claim

### 2.1 Built but UNWIRED (verified by grep: zero non-test references)

These pass tests and are referenced by no runtime code path:

| Component (task) | State | What "wired" would mean |
|---|---|---|
| `ComplianceGate` + `FatFingerCheck` + `ListCheck` + `MachineGunCheck` (10.x) | library only | the edge's order path runs `LayeredValidatorPipeline` **without any compliance gate** — the 15c3-5 pack claims the fat-finger control is IMPLEMENTED, and the class exists, but **no order ever passes through it** |
| `SmartOrderRouter`, `AlgoWheel`, `SweepStrategy` (11.11/11.12) | library only | routes go venue-direct; no SOR decision in any runtime |
| `TcaService`, `BestExAuditor` (12.10/12.14) | library only | no fills flow into TCA; the committee pack can't be produced from live data |
| `FeeEngine`/`AccruedInterest`/`NetMoney` (12.13) | library only | confirms still carry clean qty×px in the live path |
| `SurveillanceFeed` + detectors (12.15) | library only | no order/exec events are exported to it |
| `DropCopyService` (12.16) | library only | no execution is mirrored anywhere |
| `CatReporter` lifecycle half (12.12) | library only | OMS lifecycle events are not captured into CAT batches (only the MEOT trade-leg adapter is registered in tests) |
| `IoiService` (9.4), `QuoteMulticast` (9.3), `JurisdictionRouter` (12.11), `AlgoCatalog` (11.16), `MarkitServNetwork` (12.4), all eight `VenueDialects` (11.3–11.10) | library only | no network adapters, no Aeron binding instantiated, no firm homing, no FIXatdl document loaded, no dialect installed on a live gateway |

**Wired and real** (for contrast): blotter projection, kill switch, baskets, watchlist,
ESP click-to-trade, RFQ workflow (dealers/eligibility/booking), notifications, P&L,
`BenchmarkService` + `QuoteServer` + pricing marks (since the Bloomberg commit), market-data
feed selection incl. Bloomberg, SSO/SCIM, maker-checker, audit-trail endpoint, drop-in OTel.

### 2.2 Mock-only implementations (acceptable seams vs. missing logic)

- **Acceptable** (seam + real logic, wire pending external access): regulator submission wires
  (TRACE/CAT/MSRB/SDR/RTS22 build real payloads, mock the ack), `MockEspVenue`,
  `MockRfqDealer`, `FixVenueSimulator`, `ReflectiveBlpapiDriver` (real binding, needs the jar).
- **Thinner than they look**:
  - `schemas/validator-rules/*.yaml` (6.4, 185 rules) — loaded **only by the consistency
    test**. No runtime evaluates them; the per-asset validation they describe is not enforced.
  - SBE instrument schemas — codegen runs only in `ems-transport`; the 22 instrument XMLs are
    consumed by the codegen sweep but no runtime encodes/decodes instrument messages with them.
  - C++ tree (`cpp/ems-*`) — headers + a compile test; functionally a placeholder.
  - TCA decomposition (impact/timing/opportunity per arch-tca) — only slippage-vs-benchmark is
    implemented; the decomposition models are absent.
  - `QuoteServer` L2 depth — topic scheme reserves `.l2`; nothing publishes depth.

### 2.3 Honest accounting

The per-task notes in `IMPL/PLAN.md` are accurate about what was built (most say "built against
simulator" or "mock-wire scope"). The gap is the absence of a **wiring/integration phase**: no
task ever said "compose the libraries into the runtime." Recommendation: open a Phase 19 with
explicit wiring tasks (one per row of §2.1), each verified by an integration test (§1.3).

## 3. Usability & operability gaps

### 3.1 Trader safety (P0)

| Finding | Where |
|---|---|
| **No confirmation on cancel** (single or batch) — accidental right-click → Cancel(8) is instant and irreversible | `main.ts` context-menu actions |
| **Amend uses a stale snapshot** — prefill from selection time; the order may have filled in between (amend-to-0 risk persists past the QA #4 fix) | `ticket.ts:341` |
| **Kill switch uses native `prompt()`/`alert()`** (QA #10 open) — blocked in embedded browsers; no typed-confirmation ("type RELEASE") for the firm-wide control | `main.ts:712` |
| No loading/disabled state on ROUTE/STAGE buttons — double-click = double order risk | `main.ts:387` |
| Recovery watchdog health-checks a single hardcoded FIGI; if that one endpoint breaks the page reload-loops | `recovery.ts:19` |

### 3.2 Open QA backlog (from docs/QA_REVIEW.md, still true in code)

#9 baskets↔orders linking (needs server-side basket id on order rows) · #10 kill prompt ·
#11 no logout / session expiry · #13 working-order dropdown polls 1s and loses selection ·
#14 P&L TOTAL row sorts lexicographically · #15 context menus and ESP tiles are mouse-only,
no ARIA · #16 cancel-from-SENT needs a Route FSM transition.

### 3.3 Formatting / i18n (P2 but trader-facing daily)

- `.toFixed(4)` for **all** asset classes — wrong for bonds (2-3dp), futures (2dp); GBp lines
  show four decimals of pence.
- **No thousand separators** anywhere — a 5,000,000 bond face reads as `5000000`.
- Timestamps in browser-local time with no timezone label — NY and London disagree on every
  audit conversation.
- All strings hardcoded English.

### 3.4 Configuration locked in code (P1)

| What | Hardcoded at |
|---|---|
| Venue list (ticket dropdown) | `ticket.ts:17` `VENUES = ["XNAS","XNYS","ARCX"]` — FI/FX instruments offer equity venues |
| Account | single `ACC-DEMO` input; no account picker, no entitlement scoping |
| RFQ TTL (25s) + poll (700ms) | `rfq.ts` — corp bonds want minutes, equity blocks want seconds |
| Fat-finger thresholds, fee schedules, wheel weights, RFQ dealer panel, eligibility | Java constructors only — no runtime config, despite a config service existing (OPERATIONS.md §config) |
| Demo credentials (`trader-token`), firm/desk | `index.html` + `TraderDesktopEdgeMain` |

### 3.5 Operability

- Persistence contract undocumented: on edge restart, orders/RFQs/positions are **gone**
  (in-memory; the event-sourcing spine is not wired into the demo edge's recovery), while
  layouts/session survive in the browser. Nothing in the README says which is which.
- `run-trader-demo.sh`: logs to unrotated `/tmp` temp files; port-conflict error gives no
  `lsof` hint.
- No correlation id in UI↔edge logs; tracing exists (18.26) but log lines don't carry trace ids.
- OPERATIONS.md lacks runbooks for the three most common dev failures: first-build Gradle
  latency, UI-proxy-to-dead-edge behavior, WS reconnect loops.
- Docs drift: instrument/panel/test counts appear hardcoded in several docs and will rot —
  consider generating them. (An audit-agent claim that `20_workflows/` is a broken reference was
  itself wrong — the directory exists; one reminder that single-pass doc audits need spot checks.)

## 4. Recommended next steps (in order)

1. **Phase 19 "wire it" tasks** — one per §2.1 row. The pattern already exists: the edge wires
   RFQ/benchmarks/quote-server today. Add `ComplianceGate` into the order path first (it
   reverses the most serious claim/reality gap), then TCA/fees on the fill path, surveillance
   + drop-copy taps on executions, SOR behind the route path, dialects on the gateway.
2. **One `EdgeIntegrationTest`** that boots `TraderDesktopEdgeMain`'s wiring in-process and
   asserts each wired subsystem reacts to one order lifecycle — kills the composition-root
   blind spot and pins §2.1 progress.
3. **Trader-safety sprint** (§3.1): confirm-on-destructive, amend re-snapshot at submit,
   in-UI kill modal, button busy-states.
4. **Config service adoption** (§3.4): venues/accounts/thresholds/TTLs served by the edge,
   consumed by the UI at logon.
5. **UI unit-test runner** (vitest) for `recovery.ts`, `layout.ts`, `aggregates.ts`,
   `grid-actions.ts` logic, plus a concurrency stress-test pass over the OMS/blotter
   accumulators.
6. **Formatting layer**: per-asset-class decimal places, thousand separators, explicit UTC
   labels — one shared formatter, replacing `.toFixed(4)`.
