## Executive synthesis — the one thing that blocks the sale

The platform is a **genuinely capable equity + FX EMS** with a real fixed-income venue/post-trade edge. But it is **marketed and attested as far more than it runs**. Institutional due-diligence does not fail on missing features — it fails on **misrepresentation**, and this repo has three compounding layers of it:

1. **False regulatory attestation (L1-1..L1-7, worst).** `GET /api/v1/market-access` exports a SEC Rule 15c3-5 pack certifying six controls `IMPLEMENTED` — fat-finger, credit/capital limits (RiskEngine), CAT, best-ex — **none of which have a runtime call site**. The framework already ships a `DEFERRED` status (with mandatory rationale) built for exactly this, sitting unused. A CCO pulling this pack signs a false attestation. This transfers regulatory liability to the buyer → instant kill + legal exposure.
2. **Safety controls off-by-default (L1-4/L1-5, L2-4).** The compliance gate (rate-limiter, restricted/allow/watch lists) and Reg SHO locate run **only when `EMS_COMPLIANCE_GATE=1`**, which the shipped demo launcher never sets. A buyer's DD reproduces **zero blocks** on the demo that sells "the attestable controls a desk buys."
3. **Overstated scope (L1-16, L5).** "195/195 done ✅" and "7 asset classes" conflate *unit-tested library* with *wired product*: credit/commodities/crypto have no order path; SOR/algo-wheel/TCA/fees/surveillance/CAT/best-ex/IOI/jurisdiction-router are library-only.

**Fastest path to sellable:** stop misrepresenting (make the attestation honest + status table truthful — cheap, mechanical) **and** turn on the controls that already exist (compliance gate on-by-default incl. FatFinger + RiskEngine — bounded). That converts the headline claims from *false* to *true-and-narrower*, which is what survives DD. Building credit/commodities/crypto is a roadmap, not an overnight fix; the audit says say so.

### L1 — Claims vs reality (lead axis)

- **[L1-1] P0 — 15c3-5 attestation pack claims a fat-finger control no order executes.**
  - Claim: `EmsMarketAccessControls.java:44` exports evidence text "FatFingerCheck (10.2) on the compliance gate — notional ceiling with netting"; `README.md:73` marks fat-finger ✅; `README.md:77` "evidence pulled from the live services".
  - Reality: the wired `ComplianceGate` (`TraderDesktopEdgeMain.java:148-160`) instantiates `MachineGunCheck`, `ListCheck`, `ShortSaleLocateCheck` — **not `FatFingerCheck`**. No `.evaluate(` call on `FatFingerCheck` exists in any main source. The `/api/v1/market-access` pack exports the fat-finger *evidence string* while no order path runs the check.
  - Institutional impact: a **false regulatory attestation** — SEC Rule 15c3-5 evidence that misrepresents an unenforced control. Instant DD kill / legal exposure.
  - Acceptance check: an order exceeding the notional ceiling submitted through the live edge must be REJECTED; a test asserts `FatFingerCheck` is in the gate's check list and the market-access pack's evidence is derived from the same instantiated check (not a literal).

- **[L1-2/L1-4] P0 — Compliance gate is OFF by default (`EMS_COMPLIANCE_GATE=1` opt-in).** `TraderDesktopEdgeMain.java:122` gates ALL compliance (MachineGun/List/ShortSale) behind an env flag; `run-trader-demo.sh:71` never sets it. Out-of-box product performs **no compliance checks** — buyer DD reproduces zero blocks. Accept: default boot rejects a restricted-list order without the env flag.
- **[L1-3] P0 — Credit/capital pre-trade limits (RiskEngine) never execute.** `grep "new RiskEngine"` outside tests = empty; the `RiskLimits` handed to the pack is `new RiskLimits()` (empty, `TraderDesktopEdgeMain.java:393`) feeding only evidence, never the order path. Attested IMPLEMENTED (15c3-5(c)(1)(ii)). Accept: order breaching a seeded limit rejected in live demo.
- **[L1-5] P0 — Reg SHO short-sale locate OFF by default** — `ShortSaleLocateCheck` runs only inside the env-gated gate, yet the pack still exports a Reg SHO attestation as if enforced. Accept: default demo rejects a naked short w/o HTB/override; attestation not claimed unless in live route.
- **[L1-6] P0 — CAT submission unreachable** — `new CatReporter`/`new CatMockAdapter` outside tests = empty; no OMS event feeds CAT. FINRA CAT is mandatory. Attested ✅. Accept: Main-reachable CatReporter; demo order produces CAT new/route/fill.
- **[L1-7] P0 — Best-execution audit library-only** — `new BestExAuditor` outside tests = empty (referenced only by unwired SOR). MiFID RTS 28 / FINRA 5310 obligation. Accept: Main-reachable BestExAuditor evaluating a fill.
- **[L1-8..L1-15] P1 — Library-only (no runtime path):** SOR/AlgoWheel/SweepStrategy (routing intelligence — live edge is single-destination pass-through); AlgoCatalog (FIXatdl); TcaService; FeeEngine/NetMoney/AccruedInterest (no commission/net-money on any fill — blotter shows gross only); SurveillanceFeed (MAR/STOR emits nothing); post-trade STP/ConfirmationService/RegulatoryStageHandler/TraceMockAdapter (only AllocationBridge+DropCopyBridge wired); IoiService/QuoteMulticast; JurisdictionRouter (MiFID RTS 22 — blocks EU sales). ### L2 — Financial-math correctness (wired paths)

> Reassuring baseline: L2 verified most reachable math **sound** — PositionService realized-P&L signs/long-short flip, PnlService FX direction & per-currency segregation, BenchmarkService VWAP/TWAP (div-0 guarded), AllocationSplitter largest-remainder (conserves total), BorrowService locate, ShortSaleLocateCheck FIX side 5/6 gating. Fixes below are targeted, not systemic.

- **[L2-1] P1 — Amend to qty below cumQty produces negative `leavesQty` (no guard).** `InMemoryStagedOrderManager.amend():118-123` guards only `newQty<=0`, then `newLeavesQty = newQty - ctx.cumQty()` with no `newQty>=cumQty` check. Reachable: REST `AMEND_ORDERS` → ApiSurface → KillSwitchOrderGuard → this. Scenario: qty 1000, cumQty 600, amend qty 500 → leavesQty −100 accepted; `InMemoryRouteManager.route` then corrupts remaining-qty math. Accept: amend below cumQty rejected; assert no order ever has `leavesQty<0`.
- **[L2-4] P1 — Default wired path enforces neither `price>0` nor notional bounds.** `LayeredValidatorPipeline` (edge) runs SESSION+REFERENCE only; LIMITS/MARKET/ASSET_CLASS/ROUTE layers are always-pass stubs (`LayeredValidatorPipeline.java:119`); `stage()` checks only `qty>0`. Notional/fat-finger cap lives only in ComplianceGate = `EMS_COMPLIANCE_GATE=1`, off by default. Scenario: `NEW_ORDERS {qty:100, price:-5000}` staged & routable → PositionService folds px≤0 → nonsensical P&L. **Corroborates L1-2.** Accept: negative/zero-price order rejected on default edge; 10bn-notional order rejected without env flag.
- **[L2-5] P2 — Amend does non-atomic read-modify-write; loses concurrent fills.** `amend()` does `orders.get` (102) → derive from snapshot cumQty → `orders.put` (152), while `cancel`/`applyOrderFsmEvent` mutate via `compute`. A fill between get and put is overwritten → stale cumQty erases a booked fill. Compounds L2-1. Accept: interleave PartialFill+amend, assert cumQty monotonic; fix inside `orders.compute`.
- **[L2-2] P2 — WAP fold multiplies money in `double`.** `PositionService.java:148` `Math.round((double)avg*|net| + (double)px*qty)/total` — only double in the long-minor-unit money layer; loses precision past 2^53. Accept: property test vs `BigInteger`/`Math.multiplyHigh` reference across 2^53 boundary.
- **[L2-3] P2 — FX convert truncates toward zero + intermediate `long` overflow risk.** `PnlService.convert():153` `localFp*rateFp/10_000` — non-half-up bias every row; product can overflow `long` for a large book. Direction/segregation correct. Accept: round-half-up assertion on `.5` cases; `Math.multiplyHigh`/overflow guard.

### L3 — Trader-safety & usability UX

> Prior §3.1: cancel-confirmation **fixed**; stale-amend and native-dialogs **still open**.

- **[L3-1] P0 — Half-open WebSocket shows "live" forever; frozen blotter looks alive.** `stream.ts:36-62` sets `live` on `onopen`, only changes on close/error — no app-level ping/pong or staleness timer; `recovery.ts` polls only when *not* live, so never fires for a stalled-open socket. Trader trades on a frozen tape believing it live. **Highest-value fix.** Accept: WS double stops sending > staleness window → chip loses `live`, gains `down`.
- **[L3-2] P0 — Amend acts on stale snapshot** (`ticket.ts:341-393`): qty/px copied into inputs once at selection, never re-read at submit; partial fill between select and AMEND sends stale qty (amend-below-fill). Backend twin of L2-1. Accept: select qty100 → partial to leaves40 → AMEND re-reads current state or blocks.
- **[L3-3] P0 — ESP click-to-trade silently fails** (`main.ts:569-586`): fetch has no `.catch`; on dead/half-open backend the 1MM order promise rejects unhandled, no toast, trader can't tell if it went in. Accept: stub ESP POST to fail → tile shows error class within 2s.
- **[L3-4] P0 — No client-side fat-finger guard before submit** (`ticket.ts:311-320`): `preview_validate` sends `{figi}` only (no qty/px); STAGE/ROUTE send raw typed values; backend FatFingerCheck unwired (L1-1). Zero pre-trade friction on abnormal size/price. Accept: qty ≥ N× lot cap → blocking client warning, no POST until acknowledged.
- **[L3-5] P0 — Native `prompt()`/`confirm()`/`alert()` on firm-wide kill switch + 8 sites** (`main.ts:495-509` et al): blocks event loop (freezes streams), unstyleable, untestable; in Electron webviews `prompt()` returns null → emergency stop may silently abort; no typed-confirm. Accept: `grep -rnE "\b(prompt|confirm|alert)\(" src/` → 0 in production modules; DOM modal with typed-confirm.
- **[L3-6] P0 — ESP one-click sends hardcoded 1,000,000-unit order, no confirm, no in-flight lock** (`main.ts:557,569-587`): stray click = 1MM market order; not disabled during flight → double-click = double fill. Accept: double-click → exactly one POST; default size from config not literal.
- **[L3-7] P1 — Wave routing sends whole basket to market with no confirmation** (`ticket.ts:445-465`). **[L3-8] P1 — Single ROUTE has no confirm** (inconsistent w/ cancel). **[L3-9] P1 — Kill-release gives no success feedback** (ambiguous safety state). **[L3-10] P1 — Trading surfaces stay enabled while connection down** (only chips change; no TRADING SUSPENDED gate). **[L3-11] P1 — Order state is plain text; REJECTED visually identical to FILLED/WORKING** (`main.ts:617-651`). **[L3-12] P1 — Critical reject toast auto-dismisses in 4s, no persistent error surface.**
- **[L3-13/14/15] P2 — Hardcoded venue list + demo creds a client can't change** (`ticket.ts:17`, `index.html:758`); `prompt()` for basket name; context menu/modals lack ARIA/focus management.

- **[L4-3] P1 — `ems-it` "integration" suite is guard-free happy-path only** (always-pass validator, always-fill venue, always-ack reg). No reject/nack/partial-fill/kill/compliance path. Accept: concurrency test asserting returned avg always matches a committed (qty, notional) pair; likely fix = immutable accumulator snapshot. **[overlaps L2 — money correctness]**
